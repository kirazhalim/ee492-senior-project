# Grouped Validation Protocol

This note records the split-protocol check before running the next modeling
experiments. The goal is methodological honesty: if a split is flawed or
imbalanced, we should know that before interpreting model scores.

## Why This Exists

The current V3 training script uses a random record-level split. That is useful
for quick development, but it can overestimate generalization when records from
the same subject/session/context appear in both train and test.

For the next stage, especially the motion/input-source ablation work, we need
to separate:

- true subject generalization,
- session/date robustness,
- activity/context robustness,
- inspected external diagnosis on records 85-95.

## Tool

Protocol reports are generated by:

```bash
PYTHONPATH=src .venv/bin/python scripts/experiments/grouped_protocol_report.py \
  --config configs/v3.yaml \
  --max-records 85 \
  --group-by subject \
  --fold-mode leave-one-group-out \
  --include-event-counts
```

The script does not train models. It only reports fold composition, event
counts, duration, and warnings such as subject overlap or contexts present in
test but absent from train.

Generated reports:

```text
artifacts/experiments/grouped_protocol/subject_leave-one-group-out_records_085/
artifacts/experiments/grouped_protocol/subject_date_group-kfold_records_085/
artifacts/experiments/grouped_protocol/date_group-kfold_records_085/
```

## Protocols Checked

### 1. Subject Leave-One-Group-Out

This is the cleanest subject-generalization protocol because the test subject
never appears in train.

Result on records 0-84:

| Test subject | Train records | Test records | Test events | Warning |
| --- | ---: | ---: | ---: | --- |
| subject_01 | 83 | 2 | 7 | Very small test fold |
| subject_02 | 82 | 3 | 13 | None |
| subject_03 | 69 | 16 | 45 | `falsepositive` context absent from train |
| subject_04 | 21 | 64 | 172 | Several contexts absent from train |

Interpretation:

```text
Subject-level LOGO is methodologically honest but highly imbalanced.
It should be treated as a hard stress test, not as a single clean average score.
```

The largest-subject fold is especially problematic: most of the dataset is from
one subject, so training without that subject leaves only 21 records. If a model fails there, that may
mean poor subject generalization, insufficient training coverage, or both.

### 2. Subject-Date GroupKFold

This keeps each subject/date session together but allows the same subject to
appear in train and test on different dates.

Result:

```text
5 folds
10 subject-date groups
test records per fold: 12-32
warning: subject_overlap_train_test in every fold
```

Interpretation:

```text
Subject-date GroupKFold is a useful session/date robustness check, but it is not
a subject-generalization test because the same subject can appear in train and
test.
```

This protocol is more balanced than subject LOGO and useful for development,
but we should not describe it as unseen-subject performance.

### 3. Date GroupKFold

This groups by recording date only.

Result:

```text
5 folds
8 date groups
test records per fold: 12-32
warning: subject_overlap_train_test in most/every fold
```

Interpretation:

```text
Date GroupKFold checks session/date robustness, but it is weaker than
subject-date grouping because multiple subjects can share a date and the same
subject may still appear across train/test.
```

## Current Recommendation

There is no perfect split protocol with the current dataset. The honest way
forward is to report multiple views rather than hiding the limitation.

Recommended next evaluation structure:

1. Keep the existing random split only as a development baseline.
2. Use subject-date GroupKFold as the main balanced robustness diagnostic.
3. Use subject LOGO as a hard unseen-subject stress test and report fold-level
   results, not only a single mean.
4. Keep records 85-95 as inspected external diagnosis, not tuning or final
   holdout.
5. For any final claim, collect or reserve a new untouched holdout.

## Important Rule

Do not tune thresholds, smoothing, merge gaps, input modes, or model
architecture by looking at the outer test fold. If post-processing must be
selected inside grouped CV, selection should use only the training side of that
fold, ideally with an inner validation split.

For the next implementation, the safest first step is:

```text
Build a grouped evaluation runner that can train/evaluate fixed, predeclared
settings across folds and produce per-fold/per-activity/per-context reports.
```

Only after that should we compare full/audio-only/motion-only/fusion variants
as candidate modeling choices.
